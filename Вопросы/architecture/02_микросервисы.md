# Микросервисы: когда и как пилить монолит
- Делить при разных темпах релизов, независимом масштабировании, чётких границах домена; не делить без мониторинга/CI/CD.
- Границы: от бизнес‑капа и bounded contexts; у каждого сервиса своя БД.
- План (Strangler): выбрать кандидат → контракт API/события → своя БД → анти‑коррапшн слой → фича‑флаг → вырезаем из монолита.
- Коммуникации: HTTP/gRPC (команды) + события (реакции). Согласованность: Saga/Outbox.

```text
User Service (DB users)
  POST /users {email, name}
  emits event user.created

Notifications Service (DB notifications)
  consumes user.created -> send welcome email
```
