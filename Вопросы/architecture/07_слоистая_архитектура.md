# Слоистая архитектура (Layered)
Классическое разделение ответственности по слоям приложения.

## Слои
- Presentation/UI — контроллеры, представления, DTO.
- Application — use-case'ы/сценарии, оркестрация, транзакции.
- Domain — бизнес-логика, сущности/VO/доменные сервисы.
- Infrastructure — БД, очереди, API, логирование.

## Принципы
- Зависимости направлены вниз (UI → Application → Domain → Infrastructure по контрактам).
- Вверх зависимостей нет (Domain не знает про UI/Infra).
- Внешние детали оборачиваются адаптерами/репозиториями.

## Пример структуры
```
src/
  UI/Http/Controller
  Application/UseCase/CreateOrderHandler.php
  Domain/Order/Order.php
  Domain/Order/OrderRepository.php (интерфейс)
  Infrastructure/Persistence/DoctrineOrderRepository.php
```

## Плюсы / минусы
- + Понятно, просто начать; хорошо отделяет бизнес от инфраструктуры.
- – Часто вырождается в «анемичные» модели; риск утечек зависимостей между слоями.
