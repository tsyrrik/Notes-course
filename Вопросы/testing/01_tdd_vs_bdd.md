- TDD: пишем юнит‑тест → он красный → минимальный код → рефакторинг (Red–Green–Refactor). Фокус на коде и быстром фидбэке.
- BDD: описывает поведение с точки зрения пользователя/бизнеса (Given/When/Then, Gherkin). Часто применяют для UI/API и живой документации.
- Инструменты: TDD → PHPUnit, PyTest; BDD → Behat/Cucumber/SpecFlow.
- Простыми словами: TDD — «сначала тест, потом код, потом улучшение», BDD — «описываю сценарий на естественном языке, подтверждаю его тестом».

## Вопрос: BDD vs TDD

## Простой ответ
- TDD: сперва тест, он падает, пишем минимальный код, доводим до зелёного, рефакторим.
- BDD: описываем поведение на языке пользователя (Given/When/Then) и проверяем сценарий.

```php
// TDD: тест перед кодом
public function test_sum(): void {
    $this->assertSame(5, sum(2, 3));
}
```

```gherkin
# BDD: сценарий
Scenario: Пользователь успешно входит
  Given Я на странице входа
  When Я ввожу логин "user" и пароль "pass"
  Then Я вижу главную страницу
```

## Ответ
- TDD: пишем юнит‑тест → он красный → минимальный код → рефакторинг (Red–Green–Refactor). Фокус на коде и быстром фидбэке.
- BDD: описывает поведение с точки зрения пользователя/бизнеса (Given/When/Then, Gherkin). Часто применяют для UI/API и живой документации.
- Инструменты: TDD → PHPUnit, PyTest; BDD → Behat/Cucumber/SpecFlow.
- Простыми словами: TDD — «сначала тест, потом код, потом улучшение», BDD — «описываю сценарий на естественном языке, подтверждаю его тестом».

## Примеры

1. TDD: написать тест для `sum(2,3)=5`, затем реализовать функцию.
2. BDD: описать пользовательский сценарий «вход в систему» через Given/When/Then.

## Доп. теория

1. BDD часто строится поверх TDD: сначала сценарий, затем шаги проверяются тестами.
2. TDD лучше подходит для низкоуровневой логики, BDD — для бизнес‑поведения.
