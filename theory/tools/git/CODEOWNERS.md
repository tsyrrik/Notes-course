**CODEOWNERS** — это файл в репозитории, который определяет,  
**кто отвечает за конкретные части кода** и **кто автоматически получает запросы на ревью**.

---

### 1. Где лежит

Файл можно разместить в одном из стандартных мест:

```
.github/CODEOWNERS
docs/CODEOWNERS
CODEOWNERS
```

Обычно — в `.github/`, если проект на GitHub.  
В GitLab — просто в корне репозитория.

---

### 2. Формат

```
# Пример CODEOWNERS

# Владелец всего репозитория
*       @team-core

# Владелец отдельных директорий
/docs/  @tech-writers
/api/   @backend
/ui/    @frontend

# Конкретные файлы
/.env.example  @devops
/Dockerfile    @ops

# Несколько владельцев
/config/ @backend @infra
```

Каждая строка:  
`<путь или паттерн> <список владельцев>`  
— владельцы могут быть пользователями или командами (`@org/team`).

---

### 3. Как это работает

- Когда кто-то открывает **Pull Request**, GitHub/GitLab смотрит, какие файлы изменены.
    
- Если изменения затрагивают пути из `CODEOWNERS`, к PR автоматически добавляются эти пользователи или команды.
    
- Без их апрува PR нельзя смержить (если правило защиты ветки это требует).
    

---

### 4. Практические советы

- Раскладывай по **модулям или фичам**, а не по слоям:
    
    ```
    /orders/ @backend-team
    /billing/ @payments
    /ui/orders/ @frontend
    ```
    
- Если есть монорепо — держи один CODEOWNERS в корне,  
    но можно также иметь **локальные** в поддиректориях (GitHub читает их рекурсивно).
    
- Используй шаблон `*` в конце пути, чтобы покрывать подпапки.
    

---

### 5. Почему это важно

- Ускоряет ревью — нужные люди подключаются автоматически.
    
- Делает зоны ответственности прозрачными.
    
- Работает как мягкий контроль качества: без владельца фича не попадёт в main.
    
- Упрощает онбординг — новички видят, к кому идти с вопросами.
